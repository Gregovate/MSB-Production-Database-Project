# C — MSB Database — LOR Naming Data Contract (v1.1)
Last updated: 2026-02-20
Owner: MSB Production Crew (DB + Sequencing)

## Purpose
This document defines the **data contract** between Light-O-Rama (LOR) and the MSB Production Database.

The goal is to make ingestion deterministic and stable year-to-year by using **human-managed naming conventions** as the primary key instead of LOR UUIDs.

LOR UUIDs still get stored, but they are treated as **implementation details**, not identity.

---

## Definitions

### LOR Terms
- **Preview**: A LOR preview file that represents a staged view of props used to program a portion of the show.
- **Prop**: Any element in LOR that can be lit up (LOR definition).
- **Sub-Prop**: A child element inside a prop (can be automatically generated by tooling or manually created in LOR).
- **Channel Name**: The per-channel naming field used primarily for sequencing organization and ordering in LOR.

### MSB Terms
- **Stage**: A park/setup location (e.g., Food Collection, Winter Wonderland) represented by a required 2-character code.
- **Display**: A single physical buildable/storable item that MSB builds, maintains, stores, and installs.
- **DisplayKey**: The stable identifier for a Display derived from the LOR Comment field (canonical key).
- **Wiring Leg**: A physical controller/channel span for a display used during field setup.

---

## System of Record

### Wiring topology (source of truth)
- LOR + parsing outputs are the source of truth for:
  - network assignments
  - controller/channel ranges
  - wiring legs per display

### Physical asset truth (source of truth)
- Production DB is the source of truth for:
  - what a Display *is* (physical definition)
  - storage/pallet/rack assignments
  - maintenance/testing records
  - accurate light counts and attributes
  - kits, inventory, documents, infrastructure references

## Identity Rules (Critical)

### Rule 1 — Display identity comes from LOR **Comment**
- MSB treats **Prop / Comment / Display Name** as interchangeable naming for a physical display.
- The **Comment** field is the authoritative place for the Display identifier.
- The Production DB primary identifier is the **DisplayKey**, derived from Comment.

**Implication**
- Channel Name is NOT identity.
- LOR UUID is NOT identity.

### Rule 2 — DisplayKey must be unique for every physical display
- Every physical display that is built/stored/maintained must have exactly one unique DisplayKey.
- DisplayKey is stored as both:
  - `display_key_raw` (as entered)
  - `display_key_norm` (normalized key used for joins)

### Rule 3 — Channel Names are treated as mutable metadata (sequencing UX)
- Channel Names are important and should not be changed casually due to sequencing organization.
- However, Channel Names are stored as attributes only and never used as identity or join keys.

## DisplayKey Format Contract (REQUIRED)

### Canonical format
**The DisplayKey MUST start with a required 2-character Stage Code. This is NOT optional.**

Format:
`<SC>-<DisplayName>(-<Variation>)*(-<Sequence>)?(-<Color>)?`

Where:
- `<SC>` = required 2-character Stage Code (examples below)
- `<DisplayName>` = required physical display identifier
- `<Variation>` = optional token(s) like DS/PS/LH/RH/etc (repeatable)
- `<Sequence>` = optional ordering token (e.g., 01, 02, 03)
- `<Color>` = optional color token (per MSB standard list)

### Stage Code meaning
The 2-character Stage Code indicates **where the display is set up in the park**.

Examples:
- `FC` = Food Collection
- `WW` = Winter Wonderland
- `FT` = Festive Trees
- `DF` = Dancing Forest

> The authoritative list of Stage Codes is maintained in the Production DB `stage` table and mirrored in the Appendix of this document.

---

## Normalization rules (`display_key_norm`)
Normalization exists to prevent accidental duplicates caused by punctuation or spacing.

`display_key_norm` rules:
1. Trim leading/trailing whitespace.
2. Convert internal whitespace runs to a single hyphen (`-`).
3. Convert underscores (`_`) to hyphens (`-`).
4. Collapse multiple hyphens into one.
5. Remove leading/trailing hyphens.
6. Preserve case (recommended) OR standardize to upper-case (allowed), but be consistent across all ingestion runs.

**NOTE (stage code)**
- Stage Code must normalize to exactly 2 alphanumeric characters.
- If normalization produces anything else for `<SC>`, ingestion must flag an error.

## DeviceType Contract

### DeviceType = "None" (Undetermined in LOR UI)
- DeviceType "None" represents **physical inventory-only** elements.
- These items:
  - MUST be included in inventory/labels/setup documentation
  - MUST be excluded from wiring leg generation
  - MAY use `MaxChannels` as a count multiplier for quantity planning

**Implication**
- Production DB must support displays that have no wiring legs but still have:
  - storage locations
  - maintenance records
  - attributes (light type, counts, vendor, etc.)
  - kit associations

---

## Sub-Prop Contract

### Generated sub-props (tool-generated)
- Generated sub-props represent real physical wiring legs inside a display.
- Generated sub-props are allowed to share the same DisplayKey via Comment.
- Generated sub-props MUST preserve channel naming needed for field wiring views.

### Manual sub-props ("uses same channels as")
- Manual sub-props may alias existing channels and may cross physical displays.
- Manual aliasing MUST NOT create new physical channels.
- Field wiring must be based on **physical channels**, not manual aliasing.

**Implication**
- Wiring legs in Production DB should be sourced from parsing outputs that represent physical channel spans.

---

## Stage Derivation Contract

### Stage identity source
- Stage identity comes from the required Stage Code (`<SC>`) in DisplayKey.
- Preview naming may still be used to group/setup “Stage views”, but the stage code in DisplayKey is authoritative.

**Implication**
- A display can be moved to a different park area only by changing its Stage Code (and recording that change in Production DB history/aliasing).

---

## Ingestion Contract (What must be imported every run)

Each LOR import run must populate a snapshot with at minimum:
- Preview UUID + Preview name
- Prop UUID + Prop name
- Comment (raw) + derived DisplayKey (raw + normalized)
- DeviceType
- Network + Controller + StartChannel + EndChannel (wiring legs)
- Any tag fields used by existing parsing logic

**Primary join rule**
- The default join from LOR snapshot to Production Display is:
  - `display.display_key_norm = lor_snapshot.display_key_norm`

**Fallback join**
- For exceptions, Production may store explicit bindings:
  - `display_id <-> lor_prop_uuid` or other curated keys

---

## Validation / QA Rules (must run on each import)

### Required checks
1. **Missing Comment**
   - If DeviceType != "None" and Comment is empty → ERROR (breaks identity)
2. **Missing or invalid Stage Code**
   - If DisplayKey does not start with `<SC>-` where `<SC>` is exactly 2 characters → ERROR
3. **Unknown Stage Code**
   - If `<SC>` not found in Production DB `stage` table → ERROR (or WARNING if running in “discovery” mode)
4. **Duplicate DisplayKey**
   - If two physical displays resolve to the same `display_key_norm` → ERROR
5. **Wiring leg sanity**
   - Missing controller/channel range on a wired display → WARNING/ERROR depending on rules

### Recommended reports
- New DisplayKeys (not seen in prior import)
- Missing DisplayKeys (previously seen but not in this import)
- Controller/channel changes for existing DisplayKeys
- Displays with DeviceType="None" counts summary (inventory-only)

---

## Change Control

### When renaming a DisplayKey
- Renaming DisplayKey impacts:
  - storage history linkage
  - maintenance history linkage
  - wiring lookup linkage
- Therefore:
  - renames require a Production DB alias record OR an explicit “DisplayKey changed” migration entry

### Contract versioning
- Any change to normalization rules, token parsing, or validation rules requires:
  - bumping this doc version
  - recording the date and reason
  - running a diff report to detect impact
